- Observation: The Instagram Business token (`IGAA...` prefix) is rejected by `graph.facebook.com` ("Cannot parse access token", code 190) but works against `graph.instagram.com` with `Authorization: Bearer <token>`.

**[2026-07-08] — pystagram client construction**

- Action: Keep this construction pattern in `post.py`'s `make_client()`; do not let a future edit revert to defaults.
- Confidence: high

**[2026-07-08] — Media hosting for Instagram API**

- Confidence: high

### Learnings / Inefficiencies

**[2026-07-08] — GCP org blocks public bucket ACLs via CLI**
- Observation: `gsutil iam ch allUsers:objectViewer gs://<bucket>` and `gsutil acl ch -u AllUsers:R` both failed with `412 PreconditionException: One or more users named in the policy do not belong to a permitted customer` — this GCP org enforces Public Access Prevention / domain-restricted sharing at the org policy level, which blocks `allUsers` grants regardless of bucket-level settings. The bucket had to be made public through the GCP console UI instead.
- Action: Don't retry `gsutil iam ch allUsers:...` in a loop when it 412s with "permitted customer" — that's an org policy wall, not a transient error. Point the user to the console (Storage → bucket → Permissions) instead.
- Confidence: high

**[2026-07-08] — Instagram container FINISHED status is not a publish-readiness guarantee**
- Observation: Media containers reported `status_code: FINISHED` immediately (even on first poll) but `media_publish` still failed with code 9007 / subcode 2207027 ("Media ID is not available... please wait for a moment") on 4 of 6 posts in a single grid run.
- Action: Always add a short delay + retry-on-9007 around `media_publish`, even when `status_code` already reads `FINISHED`.
- Confidence: high (4/6 recurrence in one session)

**[2026-07-08] — Instagram Graph API media cannot be deleted programmatically**
- Observation: `DELETE /{media-id}` against `graph.instagram.com` returns `Unsupported delete request ... does not support this operation` (code 100, subcode 33) for every post ID tried. This matches the Graph API reference doc, which states deleting IG Media "is not supported."
- Action: Don't attempt programmatic delete for IG posts — always tell the user it must be done manually in the Instagram app (⋯ → Delete).
- Confidence: high

**[2026-07-08] — Symlinked skill dirs lose their venv on update**
- Observation: `.claude/skills/insta` is a symlink to `.agents/skills/insta`. Running `npx skills update` regenerates the real target directory, wiping any `.venv` created inside it. The venv had to be recreated mid-session after an update.
- Action: After any `npx skills update`, re-run the skill's venv bootstrap step before assuming `pystagram`/deps are present — don't assume a previously-working venv survived the update.
- Confidence: medium (seen once, but mechanism is structural — will recur on every update)
